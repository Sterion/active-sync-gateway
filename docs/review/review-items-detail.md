## AREA A — Core: State / Sync / Backend

A1. **[High]** Cross-collection premature flush of a deferred Replay rollback — `State/CollectionStateStore.cs:42` and `:66-69` (`CollectionStateStore.ValidateSyncKeyAsync`). All `<Collection>` elements of one Sync request run sequentially on ONE request-scoped `SyncDbContext` (SyncHandler.cs:147-158). The Replay path deliberately rolls the tracked entity back (`SnapshotCompressed = PreviousSnapshotCompressed; PreviousSnapshotCompressed = null`) but does NOT save (the F12 deferral: "if the round never commits, discard it"). But the key-0 path DOES `await db.SaveChangesAsync(ct)` (line 42), and any sibling collection's `CommitCollectionStateAsync` also flushes the shared tracker. So a Replay on collection A followed by a key-0 (or a normal commit) on collection B flushes A's rolled-back-but-uncommitted state with no replay generation — exactly the F12 failure the deferral prevents: A ends rolled back with `PreviousSnapshotCompressed = null`, so A's next same-key attempt re-delivers already-sent items. Collection order is client-controlled → reachable. FIX: process each collection on its own short-lived context (like DavItemMap), or hold the Replay rollback in local state and apply it to the tracked entity only immediately before that collection's own commit; do not `SaveChangesAsync` inside `ValidateSyncKeyAsync` while sibling rollbacks are pending.

A2. **[Medium]** `GetOrAddBatchAsync` drops ids for hrefs only this request created, when a concurrent unique-violation rolls back the whole batch — `State/DavItemMap.cs:97-111` (`DavItemMap.GetOrAddBatchAsync`). `await ctx.SaveChangesAsync(ct); … catch (DbUpdateException…) { Detach all; map = await ReadHrefIdsAsync(...) }`. `SaveChangesAsync` is atomic: one href's unique violation rolls back ALL staged inserts. The recovery re-reads `wanted`, which now returns ids only for hrefs a concurrent writer actually persisted; any href new to THIS request and not created by the racer is absent from the returned map → its item has no ServerId and silently drops out of that Sync round's render window. The single-item form (line 45) is immune (re-reads exactly the one contended href). FIX: after the re-read, re-stage still-missing hrefs and `SaveChanges` again (bounded retry loop), or insert row-by-row with per-row unique-violation tolerance.

A3. **[Medium]** `SaveOofAsync`/`GetOofAsync` have no concurrency token and a read-modify-write insert race — `State/SyncStateService.cs:185-212`. `OofSetting` is not in the token-stamping set (`SyncDbContext.cs` stamps only `CollectionState or LocalItem or Device`), and `SaveOofAsync` does `FirstOrDefaultAsync … if null Add … SaveChangesAsync`. Two concurrent Oof writes (phone Settings→Oof racing an `eas` CLI edit) can both read null and both `Add` → unique-index violation on `UserName` surfaces as an uncaught `DbUpdateException` → 500; or both read the same row and lost-update each other with no conflict detection. Unlike Device/CollectionState/LocalItem there is no token to detect the lost update and no unique-violation catch to make the insert an upsert. FIX: add `OofSetting` to the token set, or catch `IsUniqueViolation` and re-read like DeviceStore/DavItemMap.

A4. **[Medium]** `PersistAsync` flushes the entire unified change tracker, coupling independent mutations — `State/SyncStateService.cs:179-182` (`SyncStateService.PersistAsync`). `=> db.SaveChangesAsync(ct);`. All collaborators share one request-scoped context, so any `SaveChangesAsync` flushes EVERY tracked pending mutation, not just the caller's. A device-field save (`SaveDeviceInfoAsync`, `SetPolicyKeyAsync`, `Device.LastSyncRequestJson`) flushes an uncommitted collection rollback (A1) and vice-versa. The transaction-policy docstring (SyncStateService.cs:55-64) argues this is safe for per-collection commits but does not account for cross-concern flushes. This is the structural fragility A1 exploits. FIX: document precisely which mutations may be in-flight simultaneously, or scope volatile per-round mutations (rollbacks) so a generic `PersistAsync` cannot flush them mid-round.

A5. **[Medium]** Auth success cache admits a stale password for up to `SuccessCacheMinutes` after a *backend-side* password change — `Backend/BackendSessionFactory.cs:96-99` (`AuthenticateAsync`). `if (…SuccessCacheMinutes > 0 && _authCache.TryGetValue(cacheKey, out cached) && cached.PasswordHash == passwordHash && cached.ExpiresUtc > DateTime.UtcNow) return true;`. The positive cache is keyed by username+SHA256(password); a hit returns `true` with no backend round-trip. If the user's mail-server password is revoked/rotated, the gateway keeps admitting the old password for the cache window — only a *gateway-side* account edit clears it (via `_onSnapshotChanged`). Bounded and arguably a documented tradeoff, but worth calling out for a security path. FIX: document the window in the option help, offer a "no positive cache" mode, or bound `SuccessCacheMinutes` more tightly.

A6. **[Low]** `CompleteAccountWipeAsync` can surface a `DbUpdateConcurrencyException` as a 500 during a security-critical wipe ack — `State/DeviceStore.cs:165-176`. The catch filter is `when (added is not null && IsUniqueViolation)`, which does not cover a concurrent Device write raising `DbUpdateConcurrencyException` (Device is token-stamped). FIX: reload+re-apply on `DbUpdateConcurrencyException` around the wipe-completion save.

A7. **[Low]** `SnapshotCodec.Decompress` uses default-comparer dictionaries while the diff engine builds Ordinal maps — `State/SnapshotCodec.cs:36,42`. `return new Dictionary<string, string>()` (and the deserialize target) use the default comparer; FolderRegistry/DavItemMap use `StringComparer.Ordinal`. Equal today for numeric ServerIds, but diff correctness (echo suppression/windowing) silently depends on the comparers matching. FIX: use `StringComparer.Ordinal` explicitly in `Decompress`.

A8. **[Low]** Cache-clear on snapshot change races an in-flight `CacheVerdict`, re-populating a stale positive verdict (TOCTOU) — `Backend/BackendSessionFactory.cs:62-66, 241-256`. Request R resolves against the old snapshot; an `eas user` edit fires `SnapshotChanged` (clears caches); R then writes the verdict computed against the now-stale account. Narrow (edit must land between resolve and cache-write) but reachable. FIX: version the snapshot and stamp verdicts with the version they were computed under; reject a hit older than current.

A9. **[Low]** `EvictIdleSessionsCore` computes `activeUsers` from raw `_sessions.Keys`, including faulted/unrealized Lazy slots — `Backend/BackendSessionFactory.cs:315-320`. A never-materialized or faulted slot still contributes its user to `activeUsers`, so that user's IDLE watchers are never trimmed even with no live session. FIX: base `activeUsers` on entries where `IsBuilt(lazy)` is true.

A10. **[Low]** Faulted Lazy session slots are never swept and can wedge a (user, device) into repeated failures — `Backend/BackendSessionFactory.cs:296-302`. The eviction predicate requires `IsBuilt(lazy)` (`IsCompletedSuccessfully`), so a build that threw (transient backend outage during `CreateAsync`) leaves a faulted `Lazy<Task<…>>` that the sweep never removes and that `GetSessionAsync` re-awaits (re-throws) rather than rebuilding. Pins memory and wedges that key until restart. FIX: in `GetSessionAsync`, catch the faulted-build exception, value-compared `TryRemove` the faulted lazy, rebuild once; have the sweep remove faulted slots.

A11. **[Low]** `ValidateSyncKeyAsync` key-0 reset destroys and commits both snapshots + ledgers with no undo — `State/CollectionStateStore.cs:36-42`. Correct for a genuine re-sync, but a spurious/duplicated client key-0 wipes `SnapshotCompressed`, `PreviousSnapshotCompressed`, and both client-command ledgers and `SaveChangesAsync`es immediately — no N-1 safety here. FIX: consider deferring the key-0 wipe to the round's commit (consistent with the F12 Replay deferral) so a key-0 that never commits does not destroy live state.

A12. **[Nit]** `GatewayMetrics` session/gauge observers are set in the factory ctor but never reset on `DisposeAsync` — `Observability/GatewayMetrics.cs:173-176` / `Backend/BackendSessionFactory.cs:73-77`. A disposed factory's static observer still captures its cleared `_sessions` until the next factory overwrites it (misleads the sessions gauge under test/multi-host). FIX: null the observer in `DisposeAsync` only if it is still the one this instance set (mirroring the `_onSnapshotChanged` detach already done).

A13. **[Nit]** Per-user metric label cardinality is unbounded on the request/sync-items counters from pre-auth usernames — `Observability/GatewayMetrics.cs:105-117` (`RecordEasRequest`). With `PerUserLabels=true` (default), `activesync_eas_requests`/`activesync_sync_items` series grow with every distinct username seen, including bogus usernames from credential-stuffing before auth. FIX: emit per-user labels only for *authenticated* requests (collapse pre-auth/failed-auth to "-"), independent of the fleet-size `PerUserLabels` switch. (See also E-area metrics handling — this is the Core/observability half.)

A14. **[Nit]** `SnapshotCodec.Compress` allocates JSON bytes + gzip buffer + final `ToArray()` per commit on a multi-MB snapshot — `State/SnapshotCodec.cs:19-26`. The documented dominant steady-state cost; not a correctness issue. FIX: serialize directly into the `GZipStream` via `Utf8JsonWriter` to drop the intermediate array and the `ToArray` (pooled buffer), or accept as-is.

## AREA B — Core: Accounts / Administration / Settings / Options

B1. **[High]** Case-only-duplicate DB account rows silently collapse, last-write-wins — `Accounts/AccountStore.cs:43-46` (`AccountStore.LoadAllAsync`). `Dictionary<…> result = new(StringComparer.OrdinalIgnoreCase); … result[entry.UserName] = options;`. `AccountEntry.UserName` has a unique index on the *raw* column; under SQLite's default BINARY collation `phone1` and `Phone1` are two distinct index values, so both rows coexist. The OrdinalIgnoreCase dictionary then keeps only one arbitrarily (enumeration order), silently discarding the other user's entire override set — an auth-affecting silent drop. The B2 `ToLower()` read-fix makes upsert *find* one of the pair but does not prevent the pair existing. FIX: enforce case-folded uniqueness at the DB layer (store `UserName` lowercased, or a computed lower column carrying the unique index), or detect+warn on collision in `LoadAllAsync`.

B2. **[Medium]** README and option docs say negative `UsersRefreshSeconds` "disables live pickup"; the code coerces negative → 0 = poll every request — `Settings/ChangeStampRefreshGate.cs:33` (`ScheduleNext`). `double seconds = double.IsFinite(refreshSeconds) ? Math.Max(refreshSeconds, 0) : 0;`. README:396 and `AuthOptions.UsersRefreshSeconds` XML doc both state negative "disables (database accounts load once at startup)". After the B11 unification a negative value polls on EVERY request — the opposite, and a per-request DB point-read the operator set negative specifically to avoid. **Names the doc's position: the code contradicts two documented contracts.** FIX: honor negative-as-disable in the gate, or correct README + the XML doc + `SettingKeys` to state negative is coerced to 0.

B3. **[Medium]** Startup-impact validation can't surface a backend/user-validator brick because it drops those from the candidate config — `Administration/SettingKeys.cs:271` (`ValidateStartupImpact`). `.AddInMemoryCollection(new Dictionary { [key] = value }).Build()` then binds `ActiveSyncOptions` and runs only `ActiveSyncOptionsValidator`, which does NOT validate backend roles or `Users` (those live in `BackendConfigurationValidator`, post-DI). A catalogue write that bricks startup only via the backend/user validators is not caught — the method's "reject at write time exactly what startup would reject" contract is only partially met. FIX: document the limit explicitly, or also run the role/user validators in the impact check where a registry is available.

B4. **[Medium]** `AccountResolver.EnsureFreshAsync` and `OnRolesChanged` race on `_snapshot` with no shared lock; a stale account refresh can overwrite a newer role-aware snapshot — `Accounts/AccountResolver.cs:117` (`EnsureFreshAsync` swap, under `_refreshGate`) vs `:170` (`OnRolesChanged` swap, config-reload thread, no gate). If a role change fires while an account refresh is mid-`BuildSnapshot` (having captured the old `_rolesProvider.Current`), the refresh can complete last and overwrite the newer snapshot, leaving declared users on stale role settings until the next stamp move. The `:146` comment ("the swap is atomic, so this needs no lock") conflates atomic store with atomic read-modify-write. FIX: serialize both writes under one lock, or re-read `_rolesProvider.Current` and re-check the signature after building.

B5. **[Low]** `Users` config secrets are unsealed to plaintext and retained in the long-lived snapshot for the process lifetime — `Accounts/AccountResolver.cs:446` (`BuildOne` → `ResolveSecret` → `RoleTemplate.Password`). Unlike the master key (zeroed at :348), unsealed backend passwords sit in GC-managed strings in the immutable `_snapshot` indefinitely. They must be usable at connect time, but keeping them resident for the whole snapshot lifetime widens the memory-disclosure surface the sealing reduced. FIX: resolve backend secrets lazily at `Resolve` time from the sealed value, or document the accepted residency.

B6. **[Low]** SQLite "missing settings table" is detected by a localized message substring — `Settings/DbSettingsLoader.cs:72` (`IsMissingSettingsTable`). `ex.Message.Contains("no such table", …)`. SQLite messages are not a stable/localized-safe API; a wording change or non-English build misclassifies a fresh-install "table missing" as a genuine outage → warning + revert of restart-tier DB-only settings (ports/TLS/metrics) to POCO defaults (the B9 silent-brick this was meant to prevent). Postgres is matched robustly by SQLSTATE (:70); SQLite has no code check. FIX: match on `SqliteException.SqliteErrorCode`/`SqliteExtendedErrorCode`, or accept the message match with a best-effort comment.

B7. **[Low]** Legacy `sieve.enabled:true` upgrade produces an un-authenticable Oof row with no Host — `Administration/LegacyAccountJson.cs:108` (`TryConvert`). `oof.Provider = "sieve";` without materializing a Host; the comment admits "the defaults-to-the-IMAP-host convenience is gone", so `BuildOne`/provider validation fails → resolver skips the account fail-closed, and `UpgradeLegacyRowsAsync` may keep re-touching it. A legacy user with only `sieve.enabled` becomes un-authenticable after upgrade. FIX: on conversion drop the Oof override when no Host can be derived, or log an actionable "set Backends:Oof:Settings:Host" hint rather than a generic error.

B8. **[Low]** `AccountStore` upsert/delete `ToLower()` predicates defeat the `UserName` index and still can't see a case-variant row the BINARY index allowed — `Accounts/AccountStore.cs:57,108,132`. `a.UserName.ToLower() == login.ToLower()` is non-sargable (full scan) and does not close B1. Negligible perf on the tiny table; the correctness gap is B1. FIX: as B1, normalize stored casing so index and lookup agree.

B9. **[Low]** `SecretRedaction.SecretMarkers` omits some real secret leaf names → cleartext render in banners/`eas config list` — `Administration/SecretRedaction.cs:24`. List = `password, passwd, pwd, passphrase, secret, token, apikey, credential`. A plugin/backend leaf named `PrivateKey`, `ClientAuth`, `Signature`, `BearerToken` (only "token" matches) is not caught; "key" is deliberately excluded. Schema-declared `Secret` fields are covered by `BackendKeyValidator.IsSecretLeaf`, but the pure name-heuristic fallback (schema-less plugins) can leak. FIX: expand the marker list or lean harder on schema-declared secret typing for plugin leaves.

B10. **[Nit]** `ResolvedAccount.OrderedRoles` lazy cache is not thread-safe on a record shared across a session's concurrent requests — `Accounts/ResolvedAccount.cs:23`. `_orderedRoles ??= Roles.Values.OrderBy(...).ToList();` — two threads racing the first read each build a list (benign, transient extra allocation + momentary identity inconsistency, the thing B28 aimed to stop). FIX: build eagerly, or `LazyInitializer`/`Interlocked` if identity matters.

B11. **[Nit]** `ValidateEntry` adds `keyError` then still calls `BuildOne` with a null key, producing a redundant "sealed but no key" failure per sealed secret — `Accounts/AccountResolver.cs:260-265`. Noisy duplicate diagnostics when the encryption key won't load. FIX: short-circuit when `keyError` is set, or suppress the derived per-secret messages.

B12. **[Nit]** `LogQueryService.LevelsAtOrAbove` accepts CLI alias "critical"→"Fatal" but `DbMinimumLevel` config validation accepts only exact enum names — `Administration/LogQueryService.cs:43` vs `Options/ActiveSyncOptionsValidator.cs:74`. `eas logs -l critical` works while `eas config set …:DbMinimumLevel critical` is rejected — two notions of "level name" that disagree. FIX: share one alias-normalizing helper.

## AREA C — WebUi

C1. **[High]** Login success clears only the per-user throttle key, never the IP-wide key → a busy NAT/proxy IP can lock out all users behind it — `Auth/AuthEndpoints.cs:146` (`LoginAsync`). `throttle.RecordSuccess(userKey);`. On failure BOTH `userKey` and `addressKey` are bumped (lines 103-104, 132-133), but success removes only `userKey`. The IP-wide counter (`addressKey`, limit `IpWideFactor × MaxFailures`) accrues monotonically across mixed failed/successful logins from one egress IP, so occasional typos from a shared IP eventually 429 everyone. The portal password path (PortalEndpoints.cs:150) correctly clears its single key. FIX: `throttle.RecordSuccess(addressKey)` too on success (or don't bump the IP key on a success-following failure).

C2. **[Medium]** Backend `/test` probe is an SSRF-style reachability oracle for arbitrary operator-supplied hosts — `Api/BackendsEndpoints.cs:178` (`Map` test handler). `bool reachable = await source.ProbeReadinessAsync(ProviderSettings.FromFlat(merged), timeout.Token);`. The coarse-message mitigation collapses the failure *text*, but the boolean `reachable` itself is the oracle: an admin (the code's own note: "not always root-equivalent") can point any role at an arbitrary `Host:Port` in `request.Settings` and learn open-vs-closed/filtered for any internal address from the gateway's network position. AGENTS.md documents the probe as intentional, but for *configured* roles' credential-less reachability — **not arbitrary operator-supplied hosts** (names the doc's position). FIX: probe only the currently-stored role settings (ignore host/port overrides from the body), or gate `/test` behind a stricter capability than plain admin.

C3. **[Medium]** `DELETE /admin/api/shares` does not normalize/validate the login before matching, unlike POST → "cannot delete what I created" — `Api/SharesEndpoints.cs:60-67` (`Map` delete handler). `string href = collectionHref.Trim(); return await shares.RemoveAsync(user, href, ct);`. POST runs `AdminIdentifiers.LoginProblem`/`HrefProblem` and `user.Trim()` (lines 40-43); DELETE trims only the href, never the user, and validates neither. A grant stored under a trimmed/normalized login can't be removed by an untrimmed/differently-cased login → silent 404. FIX: trim `user` and apply the same identifier checks as POST before `RemoveAsync`.

C4. **[Medium]** OIDC signs in an *unbound* config-declared admin on a bare login-claim match (mutable `preferred_username` takeover) — `Auth/OidcLogin.cs:61-68` (`EvaluateAsync`). For a config-declared account with no `OidcSubject`, the TOFU subject-binding branch is skipped (correct), but the code still returns `Verdict(true, login, account.Options.Admin == true || claimAdmin, …)`. So a config admin the operator never manually bound authenticates on the login claim alone — exactly the mutable-`preferred_username` takeover webui.md:79-85 warns about; the doc implies such an account should not authenticate an unbound admin on claim alone (names the doc's position). FIX: refuse sign-in (or refuse the admin bit) for an unbound account whose subject cannot be recorded — require a bound subject before honoring a config-declared admin under OIDC.

C5. **[Low]** Portal `me`/`backends/meta` echo the admin-set backend `userName` verbatim to a non-admin portal user — `Api/PortalEndpoints.cs:48-57` (`me` handler). `userName = o.UserName` (line 55) is returned for every role. For a role whose credentials an admin configured (often the full mail login / a service account), the *username* is disclosed to the portal user, who is admin-forbidden from changing Provider/Enabled but can now read what an admin bound. Username is not the secret, but it is credential-adjacent topology disclosure. FIX: echo `userName` only for roles/fields the caller may self-edit, consistent with the `SelfServiceEditable` gating on `settings`.

C6. **[Low]** Session-revocation latency is `max(60s, UsersRefreshSeconds)`, but the doc promises "at most once a minute" — `Auth/SessionValidation.cs:131-133` (`ValidateAsync`). `await resolver.EnsureFreshAsync(false, ct); resolver.MergedUsers.TryGetValue(login, out account);`. `EnsureFreshAsync(false, …)` refreshes only when the stamp poll interval elapsed, so a deleted user can stay signed in beyond the 60 s check. webui.md:114 overstates the guarantee. FIX: pass `force: true` here, or document the compound latency. (Fail-open-on-fault is correct and intentional.)

C7. **[Low]** Provider name is stored verbatim (unexpected casing) though it is resolved from an OrdinalIgnoreCase dict — `Api/BackendsEndpoints.cs:261-286`. `providerName = request.Provider ?? effective.GetValueOrDefault(ProviderKey)` looked up case-insensitively, but `changes[ProviderKey] = requested` stores the raw string; if `registry.GetFor` is case-sensitive, write-time validation passes but later role resolution differs. FIX: canonicalize the provider name to the registry's stored casing before persisting.

C8. **[Nit]** `Index()` streams the SPA HTML with no cache headers, re-reading the embedded resource per request — `Setup/WebUiApplicationExtensions.cs:132-137`. The `/admin` and `/user` index roots bypass the static-file middleware's ETag/caching. Minor. FIX: serve the index through the static-file pipeline or add cache headers.

C9. **[Nit]** `settings` PUT rejects empty value before checking `HostControlledReason`, giving the wrong error for a host-controlled key — `Api/SettingsEndpoints.cs:64-67`. An empty value to `ActiveSync:Encryption:Key` says "value is required" instead of "bootstrap settings must come from the environment". FIX: check `HostControlledReason` first.

## AREA D — Backends: Common / Imap / Smtp / Local / Sieve

D1. **[High]** SMTP submission has no MaxSize preflight; oversized mail fails only after full DATA transfer, indistinguishable from a transient error — `Smtp/SmtpSubmitBackend.cs:63` (`SendAsync`). `await smtp.SendAsync(message, ct)`. MailKit exposes `smtp.MaxSize`/`Capabilities` after auth; nothing checks `mime.Length` against it, so a 30 MB message to a 10 MB server streams the whole body before the 552 reject, and the resulting `SmtpCommandException` surfaces as a generic submit failure with no size hint. FIX: after `AuthenticateAsync`, if `smtp.Capabilities.HasFlag(SmtpCapabilities.Size) && smtp.MaxSize > 0 && mime.LongLength > smtp.MaxSize`, throw a distinct non-retryable BackendException before `SendAsync`.

D2. **[Medium]** Mail listing applies a server-side `DeliveredAfter(since.Date)` filter, narrowing the "whole truth" revision map and crossing a TZ boundary — `Imap/ImapMailBackend.cs:104` (`GetItemRevisionsAsync`). `SearchQuery.DeliveredAfter(since.Date)`. Per AGENTS.md H16 the revision map is the whole truth; a server-side date filter means items outside the window are absent, which the diff reads as deletions — safe ONLY if the engine scopes the stored snapshot to the identical `SinceUtc`. The `.Date` floor also compares server-local INTERNALDATE against a UTC `since`, so items near midnight can be filtered inconsistently vs `SnapshotStatusAsync` (which never filters). FIX: confirm the engine scopes the stored snapshot to the same filter; document the INTERNALDATE-vs-UTC skew or widen by a day.

D3. **[Medium]** `ContactConverter.AppendPreserved` can re-emit EMAIL lines already written by the managed rewrite, producing duplicate EMAIL properties — `Common/Converters/ContactConverter.cs:262-279`. The managed rewrite emits `EMAIL1-3` from the merged payload (line 187-188), then `AppendPreserved` walks the *original* card and re-emits every EMAIL beyond the first 3 (`if (emailCount > 3) AppendFolded(...)`); when the original ordering differs from `OrderByPref`, a "surplus" address can duplicate one already written as Email1-3. FIX: track which addresses were emitted and suppress preserved EMAILs matching an emitted value.

D4. **[Medium]** Type-4 (full MIME) body applies the plaintext byte-cut `TruncateUtf8` to raw MIME, corrupting it — `Common/Converters/MailConverter.cs:147-151` (`BuildBody`). `content = BodyText.TruncateUtf8(content, limit);`. For a Type-4 body the content is serialized `message/rfc822`; cutting at a byte boundary mid-header/mid-part yields unparseable partial MIME (the client is told Truncated=1 to re-fetch, tolerable, but a rendering client shows garbage and base64 parts split mid-line). FIX: for type 4, don't truncate (MIME fetch is all-or-nothing) or truncate only on a part boundary.

D5. **[Medium]** All-day calendar events emit `StartTime`/`EndTime` from `AsUtc`, shifting a non-UTC-anchored all-day event to the wrong day — `Common/Converters/CalendarConverter.cs:44-52` (`ToApplicationData`). `DateTime? start = ToUtc(master.Start);`. A `DTSTART;VALUE=DATE:20260401` in a non-UTC zone becomes `2026-03-31T22:00Z`, and MS-ASCAL renders all-day StartTime as-is → phone shows Mar 31. `TasksConverter` handles this with `Nominal()`; the calendar all-day path does not. FIX: for `allDay`, emit the nominal date at midnight (like `Nominal`) rather than the UTC instant.

D6. **[Medium]** `SanitizeKeyword` collapses distinct EAS categories to the same IMAP keyword → perpetual category churn — `Imap/ImapMailBackend.cs:582-592`. `sanitized[i] = c > ' ' && c < 127 && !@"(){%*""\[]".Contains(c) ? c : '_';`. `"a b"` and `"a_b"` both become `a_b`; the server-derived category then never matches the client's original, so the revision string thrashes every Sync. FIX: accept the lossiness is inherent but store the original category text (drop non-atom categories rather than mangle), or document that non-atom categories are not round-trip-stable so the diff doesn't thrash.

D7. **[Low]** `MoveItemAsync` surfaces an unknown destination as an unhandled `FolderNotFoundException` rather than a clean status — `Imap/ImapMailBackend.cs:341-344`. `IMailFolder destination = await client.GetFolderAsync(...)`. FIX: wrap unknown-destination in `BackendException`/`BackendItemNotFoundException` for a clean MoveItems status.

D8. **[Low]** `ForceFrom` silently no-ops (submits the client's possibly-spoofed From) when `MailAddress` is unset — `Smtp/SmtpSubmitBackend.cs:27`. `if (options.ForceFrom && mailAddress is not null)`. The rewrite promise ("replace sender with the authenticated user") silently doesn't apply, with no warning. FIX: log a warning when `ForceFrom` is on but `mailAddress` is null.

D9. **[Low]** `ManageSieveClient` sends `AUTHENTICATE "PLAIN"` without checking the advertised SASL mechanisms — `Sieve/ManageSieveClient.cs:76-84` (`ConnectAsync`). FIX: allowlist to the known versions (or reject majors outside {2,12,14,16}) in `Parse`, or validate at the header call site.

W4. **[Low]** `FromBase64` field loop silently skips unknown tags rather than rejecting → a misaligned/hostile query parses "successfully" with wrong values — `Http/EasRequestParameters.cs:156-177` (`FromBase64`). The `switch (tag)` has no `default:`; unknown tags consume `length` bytes and continue. Low exploit value (fields are length-prefixed) but it hides desync as success instead of a clean 400. FIX: `default: throw new FormatException($"Unknown query field tag {tag}.");` or explicitly document tolerance.

W5. **[Low]** `ToBase64` writes device-id / device-type length as a single byte with no range check → silent truncation for values >255 bytes — `Http/EasRequestParameters.cs:217, 231`. `ms.WriteByte((byte)deviceId.Length);` (length 256 → 0). Test/client encoder, low severity, but a latent length truncation. FIX: `if (deviceId.Length > 255) throw new ArgumentException(...)`.

W6. **[Low]** Decoder accepts WBXML versions 0x01/0x02 though MS-ASWBXML mandates 1.3 and the tables assume 1.3 — `Wbxml/WbxmlDecoder.cs:80`. `if (version != 0x03 && version != 0x02 && version != 0x01)`. Widens the parser surface for no spec-required reason (harmless in practice — body encoding is identical). FIX: restrict to `version != 0x03` unless a concrete 1.1/1.2 client is known.

W7. **[Nit]** `WbxmlEncoder.WriteElement` computes `text` (lines 68-70) but uses it only in the opaque branch; the non-opaque path re-walks `element.Nodes()`, triple-enumerating the node list — `Wbxml/WbxmlEncoder.cs:68`. Minor allocation/CPU. FIX: compute `text` only inside the `isOpaque` branch.

W8. **[Nit]** `WbxmlEncoder.WriteInlineString` strips embedded NUL silently with no diagnostic — `Wbxml/WbxmlEncoder.cs:121`. `Encoding.UTF8.GetBytes(text.Contains('\0') ? text.Replace("\0", "") : text)`. Correct (NUL would truncate STR_I), but NUL from a backend is a data-corruption signal dropped silently. FIX: keep the strip; add a Trace log so silent value mangling is diagnosable.

## AREA K — Security / Crypto / Plugins / Observability / Contracts

K1. **[High]** Passphrase key-derivation uses a global fixed salt by default (rainbow-table exposure) — `Crypto/EncryptionKeyLoader.cs:28, 141-147` (`ResolveSalt`). `private static readonly byte[] DerivationSalt = Encoding.UTF8.GetBytes("ActiveSync.Encryption.KeyDerivation.v1")` and `if (string.IsNullOrWhiteSpace(options.KeyDerivationSalt)) return DerivationSalt;`. Passphrases are the DOCUMENTED path (`"Key": "<any passphrase>"`, README quick-start); unless an operator sets `KeyDerivationSalt` (unset by default, not in the quick-start), every deployment stretches its passphrase against one identical, publicly-known salt over 200k SHA-256 iters. A single precomputed table recovers the master key for every default-configured gateway that used a weak passphrase — and that key decrypts all local item content, sealed backend credentials, the TLS private key and CLI envelopes. The code comment (K45 in round 1) acknowledges the tradeoff but keeps the insecure default. **Names the doc's position** (`EncryptionOptions` "Unset keeps the historical fixed application salt") — a deliberate decision that is genuinely wrong as a *default*. Breaking: changing the derived key re-seals every passphrase-stretched `enc:v1:` value and re-keys local rows. FIX: derive a per-deployment salt with no operator action — generate a random salt on first boot and persist it in the state DB (already holds the sealed cert row), or refuse passphrase mode without `KeyDerivationSalt`.

K2. **[High]** `LocalContentProtector` AAD is not injective — a newline in the user name collides across (user, collection) pairs — `Security/LocalContentProtector.cs:122` (`Aad`). `return Encoding.UTF8.GetBytes(userName + "\n" + collection);`. The AAD binding is the sole mechanism (README §Security: "bound to the owning user + collection so rows cannot be swapped between users") preventing cross-user row swap. `"a\nb" + "\n" + "c"` and `"a" + "\n" + "b\nc"` produce identical AAD, so a user whose login contains `\n` can be made to share AAD with another (user, collection). Gateway logins arrive as HTTP Basic username (attacker-influenced) and the class never rejects `\n`. FIX: use an unambiguous encoding (length-prefix each part, or reject control chars in `userName`/`collection`). SecretValue's delimiter-free constant AAD avoids this — the two schemes are deliberately non-interchangeable, which is correct.

K3. **[Medium]** Attacker-controlled PBKDF2 iteration count has no upper bound — password-verify DoS — `Security/GatewayPasswordHasher.cs:20, 77` (`TryParse`). `private const int MinIterations = 100_000;` … `if (!int.TryParse(parts[0], out iterations) || iterations < MinIterations)`. Only a lower bound is enforced. A stored `pbkdf2$2000000000$…` forces every login verify to run 2e9 iterations, blocking a request thread for seconds. `eas user set … Password` / the WebUi accept already-prepared `pbkdf2$` values verbatim (README:417), so a lower-privilege path that can write an account row weaponizes it. FIX: clamp with an upper bound (reject `iterations > 10_000_000`) in `TryParse`.

K4. **[Medium]** Self-signed certificate 20-year validity is rejected by modern clients (Apple ≤825/≤398-day rule) — `Security/GatewayCertificateStore.cs:148` (`Generate`). `request.CreateSelfSigned(now.AddHours(-1), now.AddYears(20));`. iOS/macOS refuse server certs whose validity exceeds 825 days (398 for certs issued after 2020-09-01); Apple is the primary EAS client (AGENTS.md: "iOS especially — it is the strictest"). A 20-year self-signed leaf is hard-failed by iOS Mail even with manual trust in many versions, so the "device asks once to trust it" flow silently doesn't work on the strictest client. **Names the doc's position** (README:530 documents 20-year as intentional) — wrong for the stated primary client. FIX: cap validity at 397 days and add renewal (regenerate when within N days of expiry), or document that the self-signed path is non-functional on Apple.

K5. **[Medium]** Self-signed cert omits an IP SAN and only adds `localhost` DNS → name-mismatch for IP-addressed clients/healthcheck — `Security/GatewayCertificateStore.cs:140-144` (`Generate`). `san.AddDnsName(host); if (!host.Equals("localhost", …)) san.AddDnsName("localhost");`. SANs are DNS-only. With `PublicUrl` unset the host is `activesync-gateway` (resolves nowhere), so any client connecting by IP (the common Docker/k8s NodePort case, and phones pointed at an IP) gets `RemoteCertificateNameMismatch`; even an IP-literal `PublicUrl` adds the IP as a *DNS* name (doesn't match an IP TLS check). FIX: detect when the host parses as an `IPAddress` and add it via `SubjectAlternativeNameBuilder.AddIpAddress`.

K6. **[Medium]** Concurrent first-boot / unreadable-row regeneration can serve different fingerprints across replicas (MITM appearance) — `Security/GatewayCertificateStore.cs:55-95` (`GetOrCreateAsync`). The `DbUpdateException` insert-race path is handled, but the *unreadable existing row* path ("fall through and replace it", lines 57-62) has no concurrency guard: two replicas can each generate and PUT their own cert (`row.PfxProtected = pfxProtected` + SaveChanges, no token check on update), flip-flopping the served cert on restart — devices see a changing fingerprint, indistinguishable from an active MITM. FIX: add a RowVersion token on `ServerCertificate` and only replace when the current blob still fails to load under the same key; log loudly on replace.

K7. **[Medium]** CLI credential-bearing responses travel `/cli` in cleartext in `AllowPlaintext` mode — `Crypto/LocalCliEnvelope.cs:80-85` (`LocalCliResult.Seal`) cross-checked with LocalCliEndpoint. Response sealing is conditional on a key existing; in `AllowPlaintext` mode (no key) there is no seal, so `eas device password` / `eas user secret` echo live credentials over the `/cli` HTTP channel in the clear. AllowPlaintext is loopback-only (bounds it), but a co-located non-key-holding sidecar sharing loopback — exactly the threat LocalCliEnvelope's doc says sealing defends against — can then read credential responses. FIX: refuse credential-bearing CLI verbs (or `/cli` entirely) when no master key is configured, rather than degrading to plaintext responses.

K8. **[Medium]** `AuthThrottle._nextPruneUtc` is a non-volatile `DateTime`, torn reads / lost prune interval — `Security/AuthThrottle.cs:35, 115-117` (`Prune`). `private DateTime _nextPruneUtc = DateTime.MinValue;` … `if (now < _nextPruneUtc) return; _nextPruneUtc = now.AddSeconds(...)`. A torn 8-byte `DateTime` under concurrency/reordering can read an arbitrary time: far future → prune never runs → table fills to `MaxTrackedKeys` and new keys silently dropped; far past → prune every request → the O(n) scan the design tries to prevent. The RMW is racy too. FIX: store the interval stamp as `long` ticks with `Interlocked.Read`/`Exchange`, or guard with the same lock discipline used for entries.

K9. **[Medium]** `AuthThrottle` reads `DateTime.UtcNow` directly — not `TimeProvider`-injectable, so the security-critical rate limiter is untestable/unvirtualizable — `Security/AuthThrottle.cs:64, 86, 91, 114`. Window-expiry, retry-after and prune-interval cannot be deterministically tested. Same applies to `GatewayCertificateStore` backdating and `LocalCliEndpoint` replay window. FIX: inject `TimeProvider` and call `GetUtcNow()`.

K10. **[Medium]** `SharedCollection.Parse` splits on `LastIndexOf('|')`, corrupting hrefs that legitimately contain `|` — `Contracts/SharedCollection.cs:13` (`Parse`). `int separator = entry.LastIndexOf('|');`. DAV hrefs may contain `|`; an href like `/cal/a|b/` with no mode suffix is parsed as href=`/cal/a`, mode=`b/` → `readOnly = true` (fail-closed saves the *security* decision, correct) but the *href is truncated* to `/cal/a`, so the wrong collection (or none) is shared, silently. The fail-closed comment addresses only the RW/RO decision, not href corruption. FIX: treat the last `|` as a delimiter only when the tail is exactly `ro`/`rw`; otherwise keep the whole string as href.

K11. **[Medium]** Security-critical AES-256-GCM framing is implemented twice, independently — `Crypto/SecretValue.cs:28-87` and `Security/LocalContentProtector.cs:50-111`. Both hand-roll `nonce‖ct‖tag` (NonceSize=12, TagSize=16, `RandomNumberGenerator.Fill`, `using AesGcm`, base64+prefix) and the mirror unseal; the only real differences are the prefix (`enc:v1:` vs `v1:`) and AAD source. A framing bug fix (constant-time length handling, a v2 format, a nonce-reuse audit) must land in two assemblies or silently diverge. FIX: extract a `SealedBlob` primitive into `ActiveSync.Crypto` that owns the framing and takes `(prefix, aad, key)`; both callers use it (Core already references Crypto). Keep the distinct prefixes/AAD as arguments — the anti-interchange guarantee is preserved, the byte layout has one implementation. (This is the crypto half of S2.)

K12. **[Low]** `TransientRetry.SendHttpAsync` defaults `idempotent: true`, so a 5xx-after-a-successful-write replays a non-idempotent request → duplicate — `Contracts/TransientRetry.cs:69, 74` (`SendHttpAsync`). `bool idempotent = true` and `if (canRetry && IsTransientStatus(response.StatusCode))`. A 502/503/504 returned after the upstream already performed the write (a JMAP create that succeeded but the gateway timed out reading) replays with the unsafe default — the exact hazard the class's own doc says the gate exists to prevent, but the safe default is the unsafe value. FIX: default `idempotent` to `false`, forcing callers to opt into replay (the transport-exception path already gates on it correctly).

K13. **[Low]** `ServerCertificateValidator` disables revocation checking unconditionally on the custom-CA path — `Backends.Common/ServerCertificateValidator.cs:56` (`Validate`). `chain.ChainPolicy.RevocationMode = X509RevocationMode.NoCheck;`. With a private-PKI CA bundle configured, a revoked backend server cert is still accepted, with no opt-in for revocation. Reasonable for lab/self-signed, but silently weakens a real private PKI that publishes CRLs/OCSP. FIX: make revocation mode a knob (default NoCheck for compatibility), or document that CaCertificatePath disables revocation.

K14. **[Low]** A 32-byte-base64-decodable passphrase silently bypasses PBKDF2 and the length floor — `Crypto/EncryptionKeyLoader.cs:113-115, 149-154` (`DeriveKey`/`TryDecodeRawKey`). `byte[]? raw = TryDecodeRawKey(material); if (raw is not null) return raw;`. A human passphrase that happens to be valid base64 for 32 bytes is used directly as the AES key with NO stretching and NO minimum-length check — low-entropy input becomes the key verbatim. The comment calls this "deterministic either way" but misses that it also skips the K46 length floor. FIX: only take the raw path when the input is canonical base64 AND the operator opted in, or apply an entropy check; at minimum note the floor doesn't protect base64-shaped passphrases.

K15. **[Low]** `GatewayPasswordHasher.TryParse` accepts externally-supplied hashes weaker than the generator's own sizes — `Security/GatewayPasswordHasher.cs:94`. `if (salt.Length < 8 || hash.Length < 16)`. `Hash` always emits 16-byte salt / 32-byte hash, but parse accepts 8-byte salt / 16-byte hash, so an operator (or lower-privilege writer, cf. K3) can store a weaker-than-generated `pbkdf2$…` that still validates. FIX: enforce the generator's own `SaltSize`/`HashSize` on parse, or document the deliberate leniency.

K16. **[Low]** `LocalCliEnvelope.TryOpen` enforces only the replay *window*; single-use is a caller obligation, not enforced by the sealed-format type — `Crypto/LocalCliEnvelope.cs:60-68` (`TryOpen`). The doc promises "an envelope runs exactly once" but `TryOpen` validates nonce *presence* only; replay prevention depends on `LocalCliEndpoint` wiring a `ReplayCache` (done correctly today). A future second caller silently loses single-use. FIX: document that `TryOpen` is window-only and single-use is the caller's job, or accept an `ISet<string> seenNonces` so the contract is self-enforcing. (Window asymmetry itself is correctly implemented.)

K17. **[Low]** The serving path doesn't validate that an operator cert has a usable private key or is unexpired — `Security/TlsCertificateResolver.cs:144-162` (`LoadExternal`). Returns `X509CertificateLoader.LoadPkcs12(...)` with no `HasPrivateKey`/`NotAfter` check, so a keyless or already-expired cert loads "successfully" and Kestrel then fails opaquely at handshake time — defeating README:559's "fails startup with a clear error". FIX: after load, assert `cert.HasPrivateKey` and warn/throw when `NotAfter < UtcNow`.

K18. **[Low]** `TlsCertificateResolver.Describe` leaks a native key handle per call (public-key objects not disposed) — `Security/TlsCertificateResolver.cs:199-201` (`Describe`). `int? keySize = cert.GetRSAPublicKey()?.KeySize ?? cert.GetECDsaPublicKey()?.KeySize ?? cert.GetDSAPublicKey()?.KeySize;` — each `Get*PublicKey()` returns a fresh `AsymmetricAlgorithm` holding a native handle, none in `using`. `DescribeAsync` is reachable from the admin TLS panel / `eas tls` and can be polled. FIX: assign to disposable locals in `using` blocks before reading `KeySize`.

K19. **[Low]** `GatewayCertificateStore.Generate` can throw at first serve on an odd `PublicUrl` with no fallback — `Security/GatewayCertificateStore.cs:141` (`Generate`, `AddDnsName(host)`). If `HostFromPublicUrl` yields a value `AddDnsName` rejects (e.g. `https://[::1]/`), `Generate` throws and HTTPS never comes up, with no fallback to `FallbackHost`. Narrow but an unhandled startup death. FIX: catch the SAN build failure and fall back to `FallbackHost`, or route an IP literal to `AddIpAddress` (see K5).

K20. **[Nit]** `BackendConnection.DisposeAsync` `_disposed` flag is set without synchronization — a concurrent double-dispose can both pass the guard — `Contracts/BackendProviders.cs:112, 120-122` (`DisposeAsync`). `if (_disposed) return; _disposed = true;`. Doc promises idempotent disposal but the check-then-set is not atomic. Latent (disposal is normally single-threaded). FIX: `if (Interlocked.Exchange(ref _disposed, 1) == 1) return;` (flag as `int`).

K21. **[Nit]** Two overlapping log-scrubbers with adjacent-but-different rules — `Contracts/WireLog.cs:33` (`Payload`) and `Server/Eas/LogText.cs:10` (`Clean`). `Payload` neutralizes control chars except CR/LF/TAB; `Clean` additionally strips Unicode bidi overrides U+202A-202E / U+2066-2069. They are called side by side (`EasContext.cs:38`). The character-safety core is duplicated and the bidi-override defense exists only on the `Clean` path — a payload could still smuggle bidi overrides into a log. FIX: route both through one character classifier and extend the bidi-override rule to `Payload`. (This is the scrubber half of S6.)

K22. **[Nit]** `MergedFreeBusy` documents digit '4' = "no data" as rank 1 (above free) but no store ever emits '4' — dead precedence / masking risk — `Contracts/MergedFreeBusy.cs:6, 48-49` (`Rank`). `'4' => 1, // no data`. `Build` initializes digits to '0' and only raises them from `BusyPeriod.Kind` (1/2/3); '4' is never produced, so ranking no-data ABOVE free means a single spurious '4' would mask genuine free time. FIX: note the invariant that stores must never emit '4' (only the per-recipient status 163 path signals no-data), or drop the dead rank.

K23. **[Nit]** `DelimitedKey.Decode` validates only the part count, not that segments were well-formed escaped input — `Contracts/DelimitedKey.cs:20-23` (`Decode`). `string[] parts = value.Split('|'); if (parts.Length != expectedParts) return null;` then unescapes each. Encode is the only writer (escapes each part), so latent, but a crafted value with the right `|` count yields attacker-chosen parts. FIX: reject segments containing characters Encode would have escaped, or document the trust assumption.

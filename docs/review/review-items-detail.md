### Area A cross-cutting
- Migration hygiene is genuinely good (Sqlite/Npgsql 1:1 across 15 migrations, per-provider snapshots, correct provider types; `AddConcurrencyTokens` `Guid.Empty` backfill is safe). BUT nothing enforces lockstep — add a CI check that the two migration dirs agree on the ordered name list.
- One request-scoped DbContext doing too many jobs — A1/A10/A18 are three faces of it. Pick a policy: explicit transaction per EAS command, OR independent ops take their own context.
- The "unique-constraint race" idiom is copy-pasted 4× with drifting fidelity (3 too broad, 1 absent, 1 retries). Wants one `InsertOrRereadAsync<T>` helper + `IsUniqueViolation`.
- Optimistic concurrency applied to `CollectionState`+`LocalItem`, skipped on `Device` which needs it. `StampConcurrencyTokens` hardcodes the type set → use an `IConcurrencyStamped` marker interface.
- **Highest-value refactor in this area:** cached backend sessions have no lifetime contract. A2/A12/A24 + three fire-and-forget disposals all stem from `CompositeBackendSession` being shared mutable state with no ownership model. Same session handed to concurrent requests for one (user,device) with no lock → MailKit `ImapClient` (not thread-safe) exposed to concurrent use when a client pipelines. Introduce an `IAsyncDisposable` lease that refcounts, gates concurrent access, defers disposal.
- `MergedFreeBusy` is pure MS-ASCMD wire-format construction with no Core state dependency → belongs in `ActiveSync.Protocol`. Same argument for `CollectionDiff` (pure logic over two dictionaries, no EF).

### Area C cross-cutting
- **The CSRF design depends on an invisible coupling.** `X-EAS-WebUi` + `SameSite=Strict` is sound SPECIFICALLY because there is no `AddCors`/`UseCors` anywhere in `src/` — a custom header forces a preflight no cross-origin page can satisfy without a permissive CORS policy. Wants one shared function, not a convention re-implemented per file — same root cause as C18.
- **Credit:** no XSS sinks in ~2000 lines of hand-rolled frontend; TLS private keys structurally excluded from `TlsEndpoints`; login throttle uses separate `webui:` key namespaces so a phone retry storm and a web attack never share counters (`AuthEndpoints.cs:84-90`) and is correctly reused for the portal password change (`PortalEndpoints.cs:126`); login responses deliberately indistinguishable for unknown-user vs wrong-password (94-100); destructive device ops require a typed confirmation echo (`DevicesEndpoints.cs:87,106`); OIDC principal re-minted from scratch so IdP claims never enter the session.
- **Suggested order:** C1 first (only one exploitable from the lowest privilege level). Then C2+C4 together (same `CookieSecurePolicy` change). Then C3 (largest, and C14 falls out of it). Then C5, C6.

### Area E cross-cutting
- **Startup-time snapshot vs. live value** is the dominant theme. The architecture is "everything DB-backed, live within ~1 s", but the composition root captures options once (`ProgramServer.cs:84`) for Kestrel, metrics, the `/metrics` filter, the banner, and `PerUserLabels`. Some are legitimately restart-tier and documented; `PerUserLabels` (E33) is not, and the inconsistent `IOptionsSnapshot`-vs-`IOptionsMonitor` split (E6) shows the distinction isn't yet a conscious convention. Write it down — a comment block listing the restart-tier settings, and a rule that request-path code always uses `IOptionsMonitor`.
- **Silent-failure paths** are the second theme. E9, E11, E24-adjacent and E34 are the same shape: a catch correct in intent (don't let observability/best-effort work break the request) that leaves no trace when the failure is permanent rather than transient. The DB log sink (E9) is the worst — its failure mode is loss of the diagnostic channel itself. A consistent policy — log the first occurrence and every Nth thereafter, to `SelfLog` where the logger itself is suspect — covers all of them.
- **Long-poll deserves its own test pass.** E7 and E8 show the contract between `LongPollWatchdog` and callers isn't enforced: nothing stops a watcher completing non-positively, faulting, or ignoring cancellation, and each degrades/breaks the heartbeat differently. Add integration tests for: a watcher returning empty mid-heartbeat, a watcher throwing, a watcher ignoring its token, and `WatchdogSeconds=0`. The `MaxHeartbeatSeconds` default of 1770 s makes E1 (HTTP/2) more consequential — an h2 client gets a Ping with no parameters, falls to Status 3, and re-polls immediately in a loop.
- **Pre-authentication work on the hot path is larger than needed.** Between eager handler materialization (E4), per-request `IOptionsSnapshot` binding (E6), per-request `PolicyDocument.Hash` (E5) and per-request logger creation (E29), a request destined for 401 still pays for nearly all of it. Fixing E4+E6 cuts steady-state allocation for a polling fleet; fixing E2 removes the pre-auth metrics attack surface at the same time.
- **Verified good:** `ValidateScopes` + `ValidateOnBuild` in all environments rules out captive dependencies (the singleton graph was traced — none capture scoped). The one construct DI validation can't see, `AddTransient(sp => sp.GetRequiredService<BackendRolesProvider>().Current)`, is also safe — every `BackendRolesConfig` injection site is a request-scoped endpoint delegate. The `Setup/` extension split, the explicit reasoning about post-Build validation, consistent `LogText` sanitisation at client-string log sites, and the hash-based re-provision trigger are all above the bar.

F1. **[High]** `<Sync/>` with no `Collections` is rejected with Status 13 instead of replaying — `Handlers/SyncHandler.cs:42-49`. H7. **[High]** Update patches omit cleared fields → deletions never reach the server — `Jmap/JsContactConverter.cs:148-269`, `Jmap/JsCalendarConverter.cs:116-187`. ### Area H cross-cutting
- **The two backends are structurally parallel and should share more.** `JmapClient` and `WebDavClient` are near-twins: same `SocketsHttpHandler` setup (`AllowAutoRedirect=false`, 10-min `PooledConnectionLifetime`), same Basic-auth construction, same 100 s timeout, byte-identical `IsSafeRedirect`, and `SendFollowingRedirectsAsync` differing only in a comment (the JMAP copy says "Mirrors WebDavClient"). That duplication is why H17 (SSE timeout) exists in one and not the other, and why H1 (TLS bypass) could diverge between two probes that should be one. Extract a `BackendHttpClientFactory` into `Backends.Common` owning handler construction, TLS wiring, auth header, timeout policy, and redirect-following; leave protocol semantics in each client. Both readiness probes should call one shared helper.
- **Both backends are permanently in full-enumeration mode** (H16). No `Foo/changes`, `sync-collection`, `queryState`, or `ifInState`, despite the state tokens being available and (DAV) actually fetched for a different purpose. This single decision has the widest blast radius — it directly causes H8, H13, H14, H15 and the general absence of conditional requests. Reasonable v1, but worth an explicit decision record.
- **The JSCalendar/JSContact bridges are the least trustworthy code in scope.** Five of seven High findings (H4 H5 H6 H7 H23) share a signature: written in one shape, read in another, or lossy in a way that round-trips stably after the first pass so no test notices. The mail/DAV paths reuse mature shared converters in `Backends.Common`; these two are bespoke and appear to lack round-trip coverage. Highest-value follow-up: a property-based EAS → JSCalendar/JSContact → EAS round-trip suite over a corpus covering recurrence (nth-weekday, by-month-day), all-day, floating, cleared fields, and every mapped contact field — it would have caught H4/H5/H6/H23 mechanically.
- **Update semantics need a decision, not an inference** (H7 first). The codebase encodes two incompatible beliefs about whether JMAP `update` is patch or replace; `docs/backends.md` (contact-photo note) implies replace, RFC 8620 §5.3 specifies patch. Emitting explicit nulls for cleared managed members is correct under both and removes the ambiguity permanently. Confirm Stalwart 0.16's actual behaviour and write the answer down.
- **Genuinely well done (calibration):** the manual same-origin redirect handling with credential-leak rationale in both clients; the `AllRealOnly` idempotency gate distinguishing replayable reads from `*/set`/`import`; the deliberate omission of response bodies from exception messages to keep PII out of logs; the trace logger's "method, URI and body only, NEVER headers"; and the Axigen workarounds (always-present time-range, href-rewriting resolution) which carry live-verification dates that make them maintainable.

## AREA K — Security / Crypto / Plugins / Observability / Contracts

_GatewayMetrics_

K1. **[High]** Unauthenticated metric-label cardinality bomb (user and command) — `Core/Observability/GatewayMetrics.cs:70-77`, driven by `EasEndpoint.cs:133-138`. `RecordEasRequest` tags `eas_requests` with `command` and `user`, both stashed into `HttpContext.Items[MetricsKey]` two lines BEFORE `AuthenticateAsync`, and the middleware records in a `finally` regardless of 401/429. `LogText.Clean(…,32/128)` caps length not cardinality → an unauthenticated attacker mints unbounded distinct time series until the process OOMs; `PerUserLabels=false` collapses only user. _GatewayCertificateStore_

_SecretValue_

K51. **[Medium]** AES-GCM framing duplicated verbatim between `SecretValue` and `LocalContentProtector` — `Crypto/SecretValue.cs:28-87` vs `Core/Security/LocalContentProtector.cs:50-111`. K52. **[Low]** No key-length validation or null guard on the public API — `Crypto/SecretValue.cs:28,42`. `Seal(plaintext, key)` and `TryUnseal(…, key, …)` accept any `byte[]` → `Seal` with a 16-byte key silently produces AES-128-GCM labelled `enc:v1:` (the doc says AES-256-GCM), and a null key throws `NullReferenceException` not `ArgumentNullException`; `LocalContentProtector.CreateProtected` validates exactly this, the published plugin-facing API doesn't. K53. **[Nit]** `TryUnseal`'s error strings are safe but coarse — `Crypto/SecretValue.cs:80`. _LocalCliEnvelope_

K54. **[Medium]** Sealed CLI envelopes are replayable within the 60-second window — `Crypto/LocalCliEnvelope.cs:47-48`, with `Cli/LocalCliEndpoint.cs:33` (`AuthWindowMs = 60_000`). `Math.Abs(nowUnixMs - decoded.TimestampUnixMs) > windowMs` is the only anti-replay control; the class doc states the threat model precisely (a co-located sidecar or host-network peer that observes loopback traffic), and such an adversary can capture a sealed envelope and re-POST it verbatim for 60 s — enough to replay a destructive `eas` verb once per capture; `Math.Abs` also accepts future-dated envelopes, doubling the effective window. (Same as L6.)

K55. **[Low]** `TryOpen`'s "never throws" contract is not fully honoured — `Crypto/LocalCliEnvelope.cs:25,36-46`. _Contracts — API design_

K56. **[High]** `BackendCredentials` is a record, so its synthesized `ToString()` prints the password — `Contracts/Models.cs:6`, transitively `BackendProviders.cs:68-69` (`ResolvedRole`) and `77-81` (`BackendConnectionContext`). C# records synthesize `ToString()` as `BackendCredentials { UserName = alice, Password = hunter2 }`, and `ResolvedRole`/`BackendConnectionContext` are records containing it so their `ToString()` recursively prints the plaintext too → any `LogDebug("ctx={Context}", context)`, interpolation into an exception message, or structured-logging destructuring dumps live backend passwords into the log store. Nothing does this in-repo today, but this is a published plugin contract handed to third-party code and it is the easiest way for a plugin author to leak every user's mail password (the same family is what `ICredentialVerifier` hands plugin implementations). FIX: override `ToString()` on all three to mask `Password` (`… Password = ***`); add a test asserting the password never appears in `ToString()`; document in `docs/plugins.md` that `Password` is live plaintext and must never be logged.

K57. **[Medium]** Host-only types leak into the plugin contract — `Contracts/IBackendSession.cs:115-154`. A plugin implements `IBackendProvider`/`IBackendConnection`/`IContentStore` (+ optional capability interfaces); it never implements `IBackendSession` or `IBackendSessionFactory` — those are host composition types (`CompositeBackendSession`/`BackendSessionFactory` in Core) that aggregate connections. Publishing them in Contracts enlarges the ABI the "major versions must match" rule polices, and every future change to session composition becomes a plugin-visible breaking change for no benefit (`BackendSessionInfo` at `BackendProviders.cs:179` is likewise host/admin shape). FIX: move `IBackendSession`, `IBackendSessionFactory`, `BackendSessionInfo` to `Core.Backend`; keep the operation interfaces implemented by providers.

K58. **[Medium]** `IContentStore` is a 12-member mandatory interface with no versioning affordance — `Contracts/IContentStore.cs:10-66`. Every store must implement folder creation/renaming/deletion/moves and change-waiting even when the backend has no such concept (a read-only feed provider throws from five methods), and adding any 13th member is a hard compile break for every third-party plugin — the loader's response is to refuse to start. The codebase already demonstrates the better pattern three times in the same file (`ICalendarAttachmentSource`, `IFreeBusySource`, `IReadOnlyCollectionSource` discovered by type test). FIX: split the mandatory core (`EasClass`, `OwnsBackendKey`, `ListFoldersAsync`, `GetItemRevisionsAsync`, `GetItemAsync`) from optional capabilities (`IItemMutationSource`, `IFolderMutationSource`, `IChangeWaitSource`), and give any member that must stay a default-interface-method throwing `NotSupportedException` so the interface can grow without breaking plugins.

K59. **[Low]** `CancellationToken` is not the last parameter on `DeleteItemAsync` — `Contracts/IContentStore.cs:48`. `Task DeleteItemAsync(string folderBackendKey, string itemKey, CancellationToken ct, bool permanent = false)` — every other method puts `ct` last; this interleaves it before an optional parameter, breaking the .NET convention and any analyzer that enforces it, and the optional parameter on an interface method is itself a hazard (the default is baked into each caller's call site, so changing `false` to `true` later would silently not affect already-compiled plugins). FIX: `DeleteItemAsync(folderBackendKey, itemKey, bool permanent, CancellationToken ct)` — make `permanent` required, move `ct` last (breaking change; bundle with the next major).

K60. **[Medium]** `BackendConnection.DisposeAsync` swallows nothing, aggregates nothing, and is not idempotent — `Contracts/BackendProviders.cs:97-120`. `foreach (resource in ownedResources ?? []) switch { IAsyncDisposable/IDisposable }` — (a) if the first resource's disposal throws, every remaining resource leaks (live IMAP/HTTP sockets); (b) no idempotence guard, so a double-dispose disposes everything twice (many clients throw); (c) the Stores are never disposed even though `IContentStore` implementations routinely hold connections (the provider must remember to add each store to `ownedResources`, with no compiler help); and the `IReadOnlyList<object>` bag with runtime type-switching is untyped (a non-disposable compiles fine and silently does nothing). FIX: guard with `bool _disposed`, collect exceptions into a list and throw `AggregateException` at the end, and either type the parameter as `IReadOnlyList<IAsyncDisposable>` or auto-dispose any `IContentStore` in `stores` that is disposable.

K61. **[Medium]** `IBackendProvider.CreateConnection` is synchronous in an otherwise fully async contract — `Contracts/BackendProviders.cs:135`. `IBackendConnection CreateConnection(BackendConnectionContext context)` — every other operation is `Task`-returning, but the one method that plausibly opens a socket / performs DAV discovery / fetches a JMAP session cannot await → a provider needing I/O at connect time must block a thread (`.GetAwaiter().GetResult()`, deadlock-prone) or defer the work into every subsequent call with lazy-init plumbing; changing this later is a hard break for every plugin. FIX: `ValueTask<IBackendConnection> CreateConnectionAsync(BackendConnectionContext, CancellationToken)` NOW while the ecosystem is empty (in-repo providers wrap bodies in `ValueTask.FromResult`).

K62. **[Medium]** `SharedCollection.Parse` fails open on an unrecognized mode suffix — `Contracts/SharedCollection.cs:11-19`. `mode.Equals("ro", OrdinalIgnoreCase)` — anything that isn't exactly `ro` yields `ReadOnly = false` (read-write), so `"/cal/team|readonly"` or a typo'd `"/cal/team|r0"` silently grants write access to a shared calendar; `Validate` catches unknown suffixes but `Parse` is a public API called independently, and `SharedCollection` records also arrive from database grants which may not route through `Validate`. FIX: fail closed — default `ReadOnly = true` for any present-but-unrecognized suffix, or give `Parse` a `TryParse(string, out SharedCollection)` sibling returning false on an unknown suffix and make the permissive `Parse` internal.

K63. **[Low]** `Parse` and `Validate` duplicate the suffix-splitting logic — `Contracts/SharedCollection.cs:13-14,24-31`. Both compute `entry.LastIndexOf('|')` and slice `entry[(separator+1)..].Trim()` independently, and `Validate` even calls `Parse` then re-does the split → a change to one (e.g. accepting `|rw` in `Parse`) won't reflect in the other. K64. **[Low]** `Validate` silently skips the same-host check when `baseUrl` is unparseable — `Contracts/SharedCollection.cs:39-42`. `if (Uri.TryCreate(baseUrl, Absolute, out baseUri) && !Equals(uri.Host, baseUri.Host, …))` — when `baseUrl` is empty/malformed the whole cross-host guard is skipped and an absolute URL pointing at an attacker-chosen host validates, after which the gateway issues authenticated DAV requests (carrying the user's backend credentials) to that host. FIX: return a failure when `baseUrl` cannot be parsed but the entry is an absolute URL ("cannot verify '{entry}' against an unparseable DAV BaseUrl") — fail closed.

K65. **[Nit]** `DelimitedKey` has no argument validation — `Contracts/DelimitedKey.cs:12-15,18-24`. `Encode(params string[] parts)` throws from deep inside `Uri.EscapeDataString` on a null element, and `Encode()` with zero parts returns `""` which `Decode(value, 0)` can't round-trip. The round-trip itself is sound (`EscapeDataString` escapes both `|` and `%`, the property the doc claims). FIX: `ArgumentNullException.ThrowIfNull(parts)` + guards rejecting null elements and an empty `parts` array; add a round-trip property test.

K66. **[Low]** `ContentFilter` factories read the wall clock — `Contracts/Models.cs:33-58`. `FromMailFilterType`/`FromCalendarFilterType` call `DateTime.UtcNow` inside static factories → the returned `SinceUtc` is non-deterministic and the type can't be unit-tested without freezing time globally, and plugin authors can't construct a reproducible filter for their own tests. K67. **[Low]** `BackendItemNotFoundException` does not derive from `BackendException` — `Contracts/Models.cs:82-94`. `BackendException : Exception` and `BackendItemNotFoundException : Exception` are siblings, so `catch (BackendException)` (the idiom used throughout the host — `EndpointAuth.cs:74`, `AuthEndpoints.cs:108`, `LocalContentProtector.cs:111`) does not catch the not-found case; given the `LocalContentProtector` doc explicitly warns that confusing these two causes the sync engine to delete items from devices, the hierarchy should make the relationship explicit rather than rely on every call site remembering both. `BackendException` is also `sealed`, so plugins can't introduce typed backend errors. K68. **[Nit]** `ProviderSettings.Empty` shares one static `IConfigurationSection` — `Contracts/BackendProviders.cs:30-34`. `EmptySection` is built once from `new ConfigurationBuilder().Build().GetSection("empty")` and shared process-wide; effectively read-only so safe today, but `IConfigurationSection` exposes a settable `Value` and an `IConfigurationRoot` any plugin holding `ProviderSettings.Empty.Section` could reach and mutate, affecting every other provider that uses `Empty`. K69. **[Low]** `DescribeConfiguration` is the only versioning affordance and it is undocumented as such — `Contracts/BackendProviders.cs:147-154`. The default interface method is the right tool and the doc explains the motivation ("an out-of-repo plugin built against an older contract"), but it is the only member using the pattern, there is no `ContractVersion` constant anywhere, and `PluginLoader`'s major-version gate (K39) is the sole compatibility mechanism → a plugin author has no way to ask "what contract am I running against?" at runtime. FIX: add `public static class GatewayContract { public const int Version = 1; }` to Contracts, surface it in the startup banner alongside loaded plugins, and adopt the DIM pattern deliberately for every future addition (documented in `docs/plugins.md`).

K70. **[Low]** `IGatewayPlugin.Register` hands plugins the live host `IServiceCollection` — `Contracts/IGatewayPlugin.cs:14-17`, invoked at `PluginLoader.cs:101`. The doc frames this as "register your provider exactly like the in-repo ones", but the parameter is the whole collection: a plugin can `RemoveAll<IBackendSessionFactory>()`, replace `LocalContentProtector`, or swap the password hasher — silently, at startup, with no log line. Consistent with "plugins are trusted code", but the contract's own docs don't say so. FIX: document the trust level plainly on the interface; if a cheap guard is wanted, snapshot the descriptor identity of a few critical service types before/after each `Register` and log a Warning when a host registration was removed or replaced.

### Area K cross-cutting
- **Secret hygiene is inconsistent across the assemblies.** `AddLocalContentProtection` zeroes the derived key, `LocalContentProtector.Dispose` zeroes its copy, and `EncryptionKeyLoader`'s doc instructs callers to zero — but `TlsCertificateResolver.MasterKey()` doesn't (K14), the key material lives in unzeroable strings (K47), the TLS private key round-trips through an unzeroed `byte[]` + a base64 string (K9), and the plugin-facing `BackendCredentials` prints its password on `ToString()` (K56). The discipline exists; it just isn't applied uniformly. A short "secret handling rules" section in `AGENTS.md` plus a test that greps `ToString()` output of the credential records would lock it in.
- **Untrusted input reaches unbounded resources in three places** — attacker-controlled strings become metric labels pre-auth (K1), attacker-controlled key counts grow the throttle table with an O(n) scan per failure (K26), and attacker-sized payloads are fully copied before truncation (K33). All three are "the cap is applied one step too late". Fix shape is the same: bound at the boundary, before allocation.
- **Duplication of security-critical primitives.** The AES-GCM envelope exists twice (K51), the control-character scrubber exists twice (K36), and `SharedCollection`'s suffix parsing exists twice (K63). Each pair is currently in sync; none has a test asserting they stay that way.
- **Delimiter-joined identifiers, done right in one place and wrong in another.** `DelimitedKey` escapes each part before joining and its doc explains why; `LocalContentProtector.Aad` (K19) and `SharedCollection.Parse` (K62) both use naive delimiters. The correct primitive is already published — use it.
- **Fail-fast is a stated policy, applied unevenly.** `PluginLoader` commits to aborting startup rather than silently degrading, then warns-and-continues on a missing entry assembly (K43) and skips the version check when the reference is transitive (K39); `TlsCertificateResolver` commits to fail-fast on external certs, then doesn't validate the cert has a private key or is unexpired (K15); `GatewayCertificateStore` regenerates silently on an undecryptable row (K10).
- **Test coverage has two holes exactly where the concurrency lives.** `Core.Tests` covers the hasher, protector, key loader, secret value, plugin loader, wire log and both cert classes — but there is no `AuthThrottleTests` and no `GatewayMetricsTests`, the two files in scope with mutable shared state, locks and lifetime bugs (K26–K32, K2–K3).

### Area K — boundary verdict (Contracts / Core / Protocol / Crypto)
The extraction was the right call and the boundary largely holds — but it is under-enforced and the Protocol dependency is justified for the wrong stated reason. **What works:** `ActiveSync.Contracts.csproj` references only `ActiveSync.Protocol` + the `Microsoft.Extensions.Configuration/DependencyInjection.Abstractions` packages — no leakage of EF Core, `SyncDbContext`, `ActiveSyncOptions`, `LocalContentProtector`; the optional-capability pattern (`ICredentialVerifier`, `IPerUserResourceOwner`, `IWatcherDiagnostics`, `IReadinessSource`, `ICalendarAttachmentSource`, `IFreeBusySource`, `IReadOnlyCollectionSource`) is genuinely good design. **On Contracts → Protocol:** keep it but fix the justification — the csproj says the reference is for "EAS constants", and the only compile-time use is `EasClass.Email/Calendar` in `ContentFilter.ForClass` (`Models.cs:65-73`), called only from the host; but removing it would be wrong because plugins genuinely need Protocol (`IContentStore.EasClass`, `BackendFolder.EasClass/EasType`, and decisively `BackendItem(IReadOnlyList<XElement>)` + `IContactOperations.SearchGalAsync` require every provider to hand-build EAS XML in the correct namespaces). **Recommendations, priority order:** (1) move `ContentFilter.ForClass` + the two `From*FilterType` factories into Core and make the Protocol dependency honest (plugins need Protocol for `EasClass`/`EasFolderType` constants and for constructing ApplicationData XML); (2) move `IBackendSession`/`IBackendSessionFactory`/`BackendSessionInfo` to Core (K57); (3) fix the Crypto namespace/assembly mismatch (K49) before the package has external consumers; (4) resolve the "one package to reference" contradiction — `Contracts.csproj:11` says a plugin references only Contracts (+ optionally `Backends.Common`), but `Crypto.csproj:19-20` says plugin authors seal config secrets with `SecretValue`, and `BackendConfigField` has a `Secret` field type with nothing in Contracts to seal it — either add a minimal `ISecretProtector` to Contracts or state in `docs/plugins.md` that secret-handling plugins also take `ActiveSync.Crypto`; (5) make the boundary machine-checked — an architecture test in `Core.Tests` asserting `typeof(IGatewayPlugin).Assembly.GetReferencedAssemblies()` contains only `ActiveSync.Protocol`, `Microsoft.Extensions.*` and framework assemblies, turning a csproj comment into a build failure.

## AREA W — WBXML codec and protocol support types

_WBXML decoder (untrusted input from phones)_

W1. **[Critical]** No nesting-depth or element-count limit — memory-amplification DoS — `Protocol/Wbxml/WbxmlDecoder.cs:55-148` (the `while (!reader.AtEnd)` loop, default case 122-146). Every `0x45` byte (tag `0x05` Sync with content) produces one `XElement` and pushes `current` deeper; there is no depth counter, no element counter, no cap on total size. `ProgramServer.cs:104` sets `MaxRequestBodySize = 64 MB`, so a 64 MB body of `0x45` yields ~64M `XElement`s nested ~64M deep — multiple GB of managed heap from one authenticated request → OOM of the whole gateway. `WbxmlDecoderHardeningTests` covers lengths/entities/truncation but has NO depth/element-count test. FIX: add `depth` (inc when `hasContent` sets `current`, dec on End) and `elements` counters; throw `WbxmlException` past `MaxDepth=256` / `MaxElements=200_000` (real EAS docs are ~10-15 levels, a WindowSize-512 Sync is well under 100k); expose both as `const` and add hardening tests.

W2. **[High]** `WbxmlEncoder.WriteElement` is unbounded-recursive — StackOverflow kills the process — `Protocol/Wbxml/WbxmlEncoder.cs:40-88` (recurses at 77). `foreach (child in childElements) WriteElement(...)` with no depth guard; combined with W1, any path that echoes a client-supplied subtree into a response (Search/ItemOperations `Properties`, or future passthrough) turns a deep client document into a `StackOverflowException`, which cannot be caught in .NET → the process dies instantly, dropping every in-flight Ping/Sync. A latent crash even without a client round-trip. FIX: add a depth parameter with a hard cap (throw `WbxmlException` past ~256, matching W1), or rewrite as an explicit `Stack<>` loop.

W3. **[Medium]** `ReadMultiByteUInt` silently overflows instead of rejecting — `Protocol/Wbxml/WbxmlDecoder.cs:197-209`. `value = (value << 7) | (b & 0x7F)` runs up to 5 iterations = 35 bits into a 32-bit `uint`; bits 32-34 are shifted off silently, so `0x90 0x80 0x80 0x80 0x00` (nominally 2³²) decodes to 0, not an error. The guarded `Skip`/`ReadBytes` contain the damage today, but the parser accepts input a conformant one rejects, and any future caller inherits a silently-wrong length. FIX: `if (value > (uint.MaxValue >> 7)) throw new WbxmlException("Multi-byte integer overflows 32 bits.")` at the top of each iteration.

W4. **[Medium]** `DecodeAsync` copies an unbounded stream and truncates length to int — `Protocol/Wbxml/WbxmlDecoder.cs:28-33`. `await stream.CopyToAsync(buffer, ct)` with no size cap, then `buffer.GetBuffer().AsSpan(0, (int)buffer.Length)` — a stream larger than `int.MaxValue` makes the `(int)` cast negative/wrapped and `AsSpan` throws `ArgumentOutOfRangeException` (uncontrolled 500, not a 400). It relies entirely on the caller's Kestrel body limit; `EasContext.cs:57` repeats the pattern independently and neither enforces it. FIX: add a `long maxBytes = 64MB` parameter, read with a counting/limiting wrapper, throw `WbxmlException("Request body too large.")`; assert `buffer.Length <= int.MaxValue` before the cast. (Fixing this changes the signature — a behaviour-preserving stub can carry a red-first reproducer.)

W5. **[Medium]** OPAQUE data is base64'd into a string — 33% bloat on the mail hot path — `Protocol/Wbxml/WbxmlDecoder.cs:76-83`. `AppendText(current, Convert.ToBase64String(bytes))` — every OPAQUE blob (for SendMail/ItemOperations the entire MIME message) becomes a base64 string, so a 20 MB attachment becomes a ~27 MB UTF-16 string (~54 MB heap), which the handler then base64-decodes back to bytes: three full copies of every attachment, all LOH, per request (same round trip in reverse at encoder `:69`). FIX: attach raw bytes via `element.AddAnnotation(new OpaqueBytes(bytes.ToArray()))` and have the encoder check the annotation first (base64 text as fallback); handlers read bytes with zero transcoding. Failing that, `ArrayPool<byte>` for the encoder's intermediate.

W6. **[Medium]** Malformed base64 in an OPAQUE element escapes as `FormatException` — `Protocol/Wbxml/WbxmlEncoder.cs:69`. `Convert.FromBase64String(text)` — every other failure in this file is a `WbxmlException`, but this throws `FormatException`; since the decoder round-trips client OPAQUE data through base64 text and handlers can copy those elements into responses, a mangled value surfaces as an unrelated exception type mid-response-write (after `Response.ContentLength` may be committed). FIX: `if (!Convert.TryFromBase64String(text, buffer, out int written)) throw new WbxmlException($"Element '{element.Name.LocalName}' is marked opaque but its content is not valid base64.")`.

W7. **[Medium]** Encoder silently drops text when an element has both text and child elements — `Protocol/Wbxml/WbxmlEncoder.cs:67-85`. The `if (isOpaque) … else if (childElements.Count > 0) … else` chain is mutually exclusive: an element with children AND text emits only the children (text vanishes); an opaque element with children emits only the blob (children vanish). Nothing warns; `hasContent` at 61 even counts the text that gets discarded. Mixed content is rare in EAS but silent data loss on the wire corrupts sync invisibly. FIX: throw `WbxmlException` when `isOpaque && childElements.Count > 0`, and when `childElements.Count > 0 && !string.IsNullOrEmpty(text)` — the encoder's input is server-built, so this is a programming error worth failing loudly on. Add a round-trip test.

W8. **[Medium]** Encoder writes NUL characters into inline strings, corrupting the document — `Protocol/Wbxml/WbxmlEncoder.cs:81-84`. `WriteByte(StrI); Write(UTF8.GetBytes(text)); WriteByte(0x00)` — STR_I is NUL-terminated, so if `text` contains `\0` (possible for a subject or display name from IMAP/CardDAV) the string terminates early and the remaining bytes are reinterpreted as tokens → a structurally valid-looking but semantically scrambled document the phone mis-syncs. FIX: reject or strip (`text.Contains('\0')` → throw, or `text.Replace("\0","")` for backend-sourced data); also strip other XML-illegal control chars (< 0x20 except tab/CR/LF).

W9. **[Nit]** Dead read before an unconditional throw — `Protocol/Wbxml/WbxmlDecoder.cs:94-96`. `case StrT: _ = reader.ReadMultiByteUInt(); throw new WbxmlException("String table references are not used by EAS.")` — the read can itself throw a different message ("Unexpected end of WBXML data") on a truncated STR_T, masking the real reason, and the value is discarded anyway. FIX: drop the read; throw directly.

W10. **[Nit]** `MemoryStream` in `Encode` is never disposed — `Protocol/Wbxml/WbxmlEncoder.cs:23`. `MemoryStream output = new();` with no `using` — harmless for `MemoryStream` but inconsistent with `EasRequestParameters.ToBase64` and trips static analysis. FIX: `using MemoryStream output = new();`.

W11. **[Low]** Charset `0x03` (US-ASCII) is accepted but strings are decoded as UTF-8 — `Protocol/Wbxml/WbxmlDecoder.cs:45-47` vs `:218`. The header permits `0x03` alongside `0x6A` (UTF-8) but `ReadNullTerminatedString` unconditionally uses `Encoding.UTF8` → a document declaring ASCII with bytes ≥ 0x80 gets UTF-8 replacement behaviour rather than rejection. Harmless (UTF-8 is an ASCII superset) but the declared charset is effectively ignored. FIX: drop `0x03` from the accepted set (EAS clients always send `0x6A`), or store the charset and pick the encoding.

_Code page tables_

W12. **[Low]** Tables are correct against MS-ASWBXML — two gaps — `Protocol/Wbxml/WbxmlCodePages.cs:131-139` (Tasks), `:288-296` (Find). Pages 0-2,4-5,7-9,12,14-18,20,25 verified token-by-token; the deliberate gaps are all correct. W13. **[Low]** A duplicate tag name anywhere fails as `TypeInitializationException` at first request — `Protocol/Wbxml/WbxmlCodePages.cs:300-304`. `Reverse = Tokens.ToDictionary(kv => kv.Value, kv => kv.Key, StringComparer.Ordinal)` — two entries with the same name in one page throw `ArgumentException` inside a static initializer → a `TypeInitializationException` on the first WBXML request the gateway ever serves, not at build/test time. FIX: add a `Protocol.Tests` unit test walking `Pages` and asserting per page that token keys and tag names are unique, `Index` equals list position, and `Reverse` round-trips every token — converts a runtime landmine into a build failure (also the natural home for spot-checking against the spec).

W14. **[Nit]** Table organisation: hand-maintained C# tuples with no provenance — `Protocol/Wbxml/WbxmlCodePages.cs:31-298`. ~270 lines of `(0x05,"Sync")` with version applicability only in trailing comments (`// 16.x`), no machine-readable record of which protocol version introduced a token → version-gating lives in handler code and reconciling against a spec revision means diffing prose. FIX: move to an embedded resource (`codepages.tsv` with page/token/name/since-version), or add a `since` field to each entry; the W13 test can assert against the same source of truth.

_Protocol support types_

W15. **[High]** `EasDateTime.ToLong/ToCompact` silently shift `DateTimeKind.Unspecified` values by the server's UTC offset — `Protocol/EasDateTime.cs:9-18`. Both call `utc.ToUniversalTime()`; for `Utc` that's a no-op, for `Unspecified` it treats the value as local and subtracts the machine offset. Values round-tripped through EF Core (SQLite returns `Unspecified` unless a converter forces `Utc`) land here as `Unspecified`, so on a Copenhagen host every `DtStamp`/`StartTime`/`EndTime`/`DateReceived` on the wire is shifted 1-2 hours — a silent, timezone-dependent corruption that looks fine in UTC CI and wrong in production. FIX: `AsUtc` switch — `Utc` → value, `Local` → `ToUniversalTime()`, `Unspecified` → `SpecifyKind(value, Utc)` (honour the parameter named `utc`); use in both formatters and audit the EF model for `DateTime` properties lacking a UTC value converter.

W16. **[Medium]** `EasDateTime.Parse` throws `FormatException` on client-supplied garbage — `Protocol/EasDateTime.cs:20-34`. After four `TryParseExact` formats fail it falls through to `DateTime.Parse(value, …)`, which throws `FormatException` on anything unparsable; these values come from phone-supplied Sync/MeetingResponse payloads (untrusted input as an uncontrolled exception, the class the WBXML decoder carefully converts to 400), and the loose `DateTime.Parse` also accepts things the spec doesn't (`"3/4/2026"`), making failure culture-dependent. FIX: `TryParse(value, out DateTime)` using only the exact-format list; have `Parse` throw a dedicated protocol exception (or return a default); drop the loose fallback; add the no-Z basic form `yyyyMMdd'T'HHmmss`.

W17. **[Medium]** The base64-query version byte isn't validated, so bogus versions unlock 16.x code paths — `Protocol/Http/EasRequestParameters.cs:103-104`. `byte versionByte = Next(); version = $"{versionByte/10}.{versionByte%10}"` — a byte of 255 yields `"25.5"`, and `EasVersion.Parse` returns `(25,5)`, satisfying every `>= V160`/`>= V161` gate → a client can claim a version the gateway never negotiated and get 16.1-only behaviour (drafts, `AccountOnlyRemoteWipe`, structured Location) without advertising support. The command code right below is range-checked (`:107`), making the omission look accidental. FIX: validate against the supported set (`{25,120,121,140,141,160,161}`) right after parsing — reject with `FormatException` (→ 400) or clamp to the highest supported.

W18. **[Medium]** `ToBase64` silently truncates length prefixes for fields over 255 bytes — `Protocol/Http/EasRequestParameters.cs:191-192,205-217`. `ms.WriteByte((byte)deviceId.Length)` and in `Param` `ms.WriteByte((byte)bytes.Length)` — an over-long `DeviceId`/`LongId`/`AttachmentName`/`CollectionId` wraps modulo 256 and produces a query the parser desyncs on, with no error at the bug site (`LongId`/`AttachmentName` are realistically long in Exchange-style deployments). This is the test-client half of the round trip, so it shows up as a mysterious integration-test parse error. FIX: `if (bytes.Length > 255) throw new ArgumentException($"'{tag}' value is {bytes.Length} bytes; the packed query allows 255.")` in both places; add a 300-byte `LongId` round-trip test.

W19. **[Low]** `ToBase64` throws raw `FormatException`/overflows on a malformed `ProtocolVersion` — `Protocol/Http/EasRequestParameters.cs:182-184`. `int.Parse(versionParts[0])` throws `FormatException` for `"abc"`/`""`, and `(byte)(major*10+minor)` wraps silently for `major >= 26` — the mirror of W17 on the encode side. FIX: use `EasVersion.Parse(ProtocolVersion)` (tolerant, already available) and range-check the resulting byte before the cast.

W20. **[Low]** `EasVersion.Parse` uses the current culture for integer parsing — `Protocol/EasVersion.cs:22-23`. `int.TryParse(value.AsSpan(0, dot), out int major)` — the two-arg overload uses `NumberStyles.Integer` + `CurrentCulture`; wire-format parsing must never depend on host locale (a culture with a different negative sign or digit substitution parses differently than on a dev machine). The rest of the protocol layer passes `InvariantCulture` explicitly. FIX: `int.TryParse(span, NumberStyles.None, CultureInfo.InvariantCulture, out int major)` for both components (`None` also rejects `"-1.4"`, which currently parses to major -1).

W21. **[Nit]** `EasFolderType` is missing types 18 and 19 — `Protocol/EasFolderType.cs:5-22`. MS-ASCMD defines 18 = Unknown, 19 = RecipientInformationCache; the enum stops at 17. Similarly `EasClass` (`EasClass.cs:6-10`) omits `"SMS"`, a valid EAS class some clients send. FIX: add `Unknown = 18`, `RecipientInformationCache = 19`; add `SMS` to `EasClass` (or document why it's deliberately unsupported).

### Area W cross-cutting
- **The WBXML layer is in good shape structurally — its gap is a specific one.** The `SpanReader` bounds-checking is careful (the comment about the uint→int cast shows real thought), the entity guard at `:89` is exactly right, and `WbxmlDecoderHardeningTests` covers lengths/truncation/code points. What's missing is the *structural* dimension of hostile input: nothing limits how many elements or how deep. **W1 and W2 together are the highest-value fixes in this area** — ~15 lines closing an OOM and a StackOverflow, both of which take down every user's sync — do them first with the corresponding hardening tests.
- **Round-trip fidelity has three silent-data-loss edges** — W7 (mixed content), W8 (NUL termination), and empty-string-vs-content-less elements. None show up in the existing round-trip tests, which construct well-formed documents. A property-based test that generates random documents from the code page tables and asserts `Decode(Encode(d)) ≡ d` catches all three, and is a natural companion to the table-validation test in W13.
- **Protocol support types are the sneakiest** — W15 (`Unspecified` shift) is a production-only corruption that UTF/UTC-clean CI never sees, and W17/W18 are the encode/decode mirror of the same "unvalidated length/version byte" mistake. The `EasDateTime`/`EasRequestParameters` pair deserves a focused round-trip + boundary test pass.

## AREA L — CLI

_Security and the `/cli` endpoint_

L22. **[High]** With no encryption key configured, any loopback process gets full admin — `Cli/LocalCliEndpoint.cs:75-82`. The sealed-envelope proof is the documented auth boundary, but when `EncryptionKeyLoader.TryLoadKey` returns null the endpoint falls back to loopback-only (`if (key is null) { args = request?.Args ?? []; … return true; }`); nothing distinguishes "dev/test" from "production where the operator fumbled the key mount". In that state any co-located sidecar, host-network peer, or unprivileged local process can run `eas user set <login> Password …`, `eas device password` (prints escrowed recovery passwords), `eas purge user --yes`, `eas config set`. The degradation is silent — the key is resolved once at Map time (`:47`) with the error discarded (`out _`). FIX: gate the fallback on `options.Encryption.AllowPlaintext == true` + a startup warning; if neither key nor `AllowPlaintext`, return 404 from `/cli` unconditionally. Stop discarding the `TryLoadKey` error at `:48`.

L23. **[Medium]** Responses (including secrets) travel loopback in plaintext while requests are sealed — `Cli/LocalCliEndpoint.cs:60-62`. `Results.Json(response)` — the request is AES-256-GCM sealed and replay-bounded, but the response is plain JSON, and `eas device password`/`eas show` return decrypted secrets in that body. The threat model that justifies sealing the request (a loopback peer that observes traffic but lacks the key) is exactly the one that can read the response. FIX: seal the response with the same key when the request arrived sealed (`Results.Text(SecretValue.Seal(Serialize(response), key))`) and have the slim client unseal it.

L24. **[Medium]** No audit trail for any `/cli` command — `Cli/LocalCliEndpoint.cs:50-63,101-108`. Commands that delete every trace of a user, disclose a device recovery password, change a gateway password, or block/unblock logins execute with no log record; the body is deliberately never logged (correct — it carries secrets), but the fact of the invocation isn't logged either, so after an incident there is nothing to reconstruct. FIX: in `ExecuteAsync`, emit one structured Information event per invocation with the verb and non-secret arguments only (`args[0]`, `args[1]`, exit code, duration; everything from `args[2]` redacted), routed through the DB log sink so `eas logs` surfaces it.

L25. **[Medium]** Process-global console redirection swallows gateway logs into the CLI response — `Cli/LocalCliEndpoint.cs:128-130,148`. `Console.SetOut/SetError/SetIn(...)` mutate process-global state inside a live web server; Serilog's console sink resolves `Console.Out`/`Console.Error` per emit, so for the duration of every forwarded command (a purge can be seconds) all gateway log output for all concurrent requests is captured into that command's Stdout and disappears from the container log. The Gate serialises CLI commands against each other but cannot serialise them against the gateway's own request handling. FIX: the `IAnsiConsole` swap is already per-request; once commands that write raw `Console` are converted to `Terminal` (L33), drop `Console.SetOut/SetError` entirely and keep only `Console.SetIn`. Short of that, an `AsyncLocal<TextWriter>`-backed `Console.Out` scoped to the async flow.

L26. **[Low]** `CapturingRegistrar.Resolve` has no cycle detection — `Cli/LocalCliEndpoint.cs:194-218`. `ctor.GetParameters().Select(p => Resolve(p.ParameterType))` recurses through constructor parameters with no visited-set and no depth cap → a type whose longest constructor takes itself (directly or transitively) produces infinite recursion → `StackOverflowException` → process death, in the gateway process; it also constructs any concrete type reflectively, so a typo'd registration silently produces a half-built object. FIX: thread a `HashSet<Type> resolving` and throw `InvalidOperationException($"Circular dependency resolving {type}")` on re-entry; cap depth at ~16.

L27. **[Low]** Envelope replay is bounded by time but not by nonce — `Cli/LocalCliEndpoint.cs:33,84-94`. `AuthWindowMs = 60_000` and `TryOpen` checks only `Math.Abs(now - timestamp) > windowMs`, so a captured envelope can be replayed any number of times within its 60 s window; for idempotent verbs that's noise, for `eas purge device`/`eas user remove` a replay re-executes the deletion, and for `eas device password` it re-discloses. Requires loopback capture (root/CAP_NET_RAW) — defence-in-depth. FIX: add a nonce to `LocalCliEnvelope`, keep a `MemoryCache` of seen nonces with a 2×window TTL, reject on hit; shrink the window to ~15 s.

_Output rendering and secret handling_

L28. **[Medium]** `eas tls` passes certificate fields through Spectre's markup parser — crashes on `[` — `Cli/TlsCommand.cs:42-54`. `table.AddRow("Subject", info.Subject)` uses the `params string[]` overload = `Markup`, not text; `Subject`/`Issuer`/joined SANs/`Fingerprint`/`CertificatePath` all originate outside this code, and a subject containing `[` (legal in an X.500 DN, or a path like `/certs/[prod]/tls.pfx`) throws `InvalidOperationException` out of the markup parser and the command dies with a stack trace. L30. **[Medium]** `eas config list` prints non-Password backend secrets in the clear — `Cli/ConfigCommands.cs:107-113`, driven by `SettingKeys.cs:196-198`. L31. **[Medium]** `eas users` state lookups are case-sensitive while the user set is case-insensitive — `Cli/InspectCommands.cs:89` vs `131-138`. L32. **[Medium]** `eas purge user` leaves shared-calendar grants behind — `Cli/PurgeCommands.cs:70-89`, help at `CliApp.cs:83-84`. L33. **[Medium]** Errors bypass the injected console entirely — throughout (`UserCommands.cs:55-57,90,120-121`; `InspectCommands.cs:328`; `LogsCommand.cs:41,51-52`; `ConfigCommands.cs:139,169,178,193`). L34. **[Low]** `device wipe` and `block` are destructive but have no confirmation — `Cli/DeviceCommands.cs:109-119`, `Cli/BlockCommands.cs:39-47`. `PurgeCommand` has a well-built confirm-or-`--yes` flow (with a non-interactive guard), but `eas device wipe` (arms a directive that removes the account from a live phone) and `eas block` (403s a user's every login) get none — both are one typo'd argument from the wrong user, and `wipe` is the most alarming verb in the CLI. _Correctness, robustness, performance_

L35. **[Medium]** Every CLI command rebuilds a full DI container, EF model and plugin set — `Cli/InspectCommands.cs:30-36`, `ConfigCommands.cs:27-34`, `LogsCommand.cs:58-63`, `TlsCommand.cs:24-27` → `CliServices.cs:17-97,106-162`. `CliServices.TryCreateAsync()` builds a new `ServiceCollection`, registers providers, `PluginLoader.LoadInto`, `BuildServiceProvider()`, `BackendConfigurationValidator.Validate()`, and `GetPendingMigrationsAsync()` (a DB round-trip + full EF model build) — per invocation; when forwarded to the warm gateway this ignores the host's already-configured provider/model/plugins. ### Area L cross-cutting
- **The CLI's Spectre-markup discipline is 90% right, which is the dangerous percentage.** `InspectCommands` has a purpose-built `AddRow` wrapping cells in `Text`, `ShareCommands` uses `Markup.Escape`, `ConfigCommands` uses `Text` + fixed-token `Markup`, `LogsCommand` uses `Text` + a whitelist. Exactly one file — `TlsCommand` (L28) — reaches for the raw `params string[]` overload, and it happens to handle the least-controlled strings (X.500 DNs and filesystem paths). The lesson isn't "escape more" — it's that the safe helper must be the *only* reachable path. Lift `AddRow` to a shared static and the class of bug disappears.
- **Secret redaction is applied per-call-site rather than per-value**, and one call site was missed (L29) plus one heuristic is too narrow (L30). Consider a `SecretString` wrapper whose `ToString()` returns `***` so leaking one requires an explicit `.Reveal()` — turns "remember to mask" into "remember to unmask", which fails safe.
- **The `/cli` auth model is well-reasoned and honestly documented, with one soft spot** — the fallback triggers on key absence rather than an explicit dev-mode flag (L22), so a production misconfiguration degrades silently to the model the design explicitly rejects. Gate on `AllowPlaintext` and log loudly. L23 (plaintext responses) and L24 (no audit) are the natural follow-ups.
- **Case sensitivity is inconsistent across the whole CLI** (L31). Logins are unioned case-insensitively, matched ordinally in memory, and compared with whatever collation the DB provider defaults to. One decision — normalise on write, or comparer-everywhere — applied across `InspectCommands`, `BlockCommands`, `PurgeCommands`, `ShareCommands`, `DeviceCommands` at once, plus a test that a mixed-case login round-trips through every verb.
- **Testing gap worth naming:** because errors go to raw `Console.Error` (L33), the `CommandAppTester` suites structurally cannot assert on any error message or exit code — the entire failure surface of an admin CLI. Fixing L33 unlocks that testing and removes two-thirds of the process-global console redirection in L25.
- **Verdict — should the CLI be split out of `ActiveSync.Server`? No**, not as framed, and the assembly boundary isn't the real problem. A clean split is blocked in three places (`ServeCommand` calls `Program.RunServerAsync`; `LocalCliEndpoint` maps into the `WebApplication` and needs the command tree in-process for the warm-start benefit; `CliServices`/`CliVerbs` depend on `Server.Setup`).

### Area A cross-cutting
- Migration hygiene is genuinely good (Sqlite/Npgsql 1:1 across 15 migrations, per-provider snapshots, correct provider types; `AddConcurrencyTokens` `Guid.Empty` backfill is safe). BUT nothing enforces lockstep — add a CI check that the two migration dirs agree on the ordered name list.
- One request-scoped DbContext doing too many jobs — A1/A10/A18 are three faces of it. Pick a policy: explicit transaction per EAS command, OR independent ops take their own context.
- The "unique-constraint race" idiom is copy-pasted 4× with drifting fidelity (3 too broad, 1 absent, 1 retries). Wants one `InsertOrRereadAsync<T>` helper + `IsUniqueViolation`.
- Optimistic concurrency applied to `CollectionState`+`LocalItem`, skipped on `Device` which needs it. `StampConcurrencyTokens` hardcodes the type set → use an `IConcurrencyStamped` marker interface.
- **Highest-value refactor in this area:** cached backend sessions have no lifetime contract. A2/A12/A24 + three fire-and-forget disposals all stem from `CompositeBackendSession` being shared mutable state with no ownership model. Same session handed to concurrent requests for one (user,device) with no lock → MailKit `ImapClient` (not thread-safe) exposed to concurrent use when a client pipelines. Introduce an `IAsyncDisposable` lease that refcounts, gates concurrent access, defers disposal.
- `MergedFreeBusy` is pure MS-ASCMD wire-format construction with no Core state dependency → belongs in `ActiveSync.Protocol`. Same argument for `CollectionDiff` (pure logic over two dictionaries, no EF).

### Area C cross-cutting
- **The CSRF design depends on an invisible coupling.** `X-EAS-WebUi` + `SameSite=Strict` is sound SPECIFICALLY because there is no `AddCors`/`UseCors` anywhere in `src/` — a custom header forces a preflight no cross-origin page can satisfy without a permissive CORS policy. Good, cheap design, but adding CORS for any reason silently removes the primary CSRF defense. Deserves a comment on the filter and an assertion in the test suite (C19). The filter correctly applies to `AllowAnonymous` endpoints too (endpoint filters run independently of authorization), so `login` is covered.
- **Recurring theme: "checked once, never again."** C3/C13/C14 are the same shape — a careful check at one moment treated as durable. The gateway is otherwise built around live re-reads (`IOptionsMonitor` everywhere, `EnsureFreshAsync` before nearly every resolver access, ~1s settings propagation). Web sessions are the one place the convention wasn't applied, and it's where staleness means "revoked admin still has admin".
- **Secret masking right in one place, absent in two** — `BackendsEndpoints.Describe` is the model; `SettingsEndpoints.Describe` does the catalogue equivalent; `UsersEndpoints.ToDto` and the `me` projection hand-roll DTOs masking only `Password` (C5). Wants one shared function, not a convention re-implemented per file — same root cause as C18.
- **Credit:** no XSS sinks in ~2000 lines of hand-rolled frontend; TLS private keys structurally excluded from `TlsEndpoints`; login throttle uses separate `webui:` key namespaces so a phone retry storm and a web attack never share counters (`AuthEndpoints.cs:84-90`) and is correctly reused for the portal password change (`PortalEndpoints.cs:126`); login responses deliberately indistinguishable for unknown-user vs wrong-password (94-100); destructive device ops require a typed confirmation echo (`DevicesEndpoints.cs:87,106`); OIDC principal re-minted from scratch so IdP claims never enter the session.
- **Suggested order:** C1 first (only one exploitable from the lowest privilege level). Then C2+C4 together (same `CookieSecurePolicy` change). Then C3 (largest, and C14 falls out of it). Then C5, C6.

**Verified CORRECT (do not "fix"):** MailKit thread-safety is handled properly — `ImapClient` never shared (`ImapSession` owns one, serializes behind `_gate` `:20,76`; IDLE watcher builds its own `ImapIdleWatcher.cs:128`; credential probe uses a throwaway `ImapBackendProvider.cs:101`); `SmtpClient` per-send. **The recurring theme in the real bugs is ASYMMETRY** — a discipline established in one place and not carried to its siblings: Expunge scoping, master-VEVENT selection, presence-guarded merges (Calendar/Tasks/Notes do it, Contacts doesn't), null-vs-throw on unparsable content, per-operation timeouts (IMAP/SMTP have them, Sieve doesn't), `EasDateTime.Parse` guarding, UTF-8-boundary truncation (`BodyText` right, `AppendFolded`/`Limit` wrong). Most would be caught by extracting the established pattern into a shared helper rather than by more review.

**Assembly verdict:** per-protocol split (Imap/Smtp/Sieve/Local) is RIGHT — keep Smtp and Sieve separate despite their size; they fill different ROLES (MailSubmit, Oof), are independently assignable per account, and provider-per-assembly is what makes them optional. Size is the wrong metric.
**But `Backends.Common` violates the boundary `ActiveSync.Contracts` was just extracted to establish.** Its csproj describes it as the reusable library "published so plugins can reuse the converters", yet it takes a hard `ProjectReference` on `ActiveSync.Core` — so a plugin author wanting `MailConverter` pulls in EF Core, the state DB, options binding, observability. The dependency is thin and removable:
  - `BackendException`/`BackendItemNotFoundException` already live in Contracts (`Models.cs:82,94`) — the converters' `using ActiveSync.Core.Backend` for them is STALE after the Contracts extraction and can simply be deleted.
  - `BackendSchemaFields.cs` needs `BackendConfigField`/`IBackendProvider` from `Core.Backend`.
  - `MailKitWireLogger` needs `ActiveSync.Core.Logging.WireLog` — one static helper.
  - `TransientRetry` is in `Core.Backend`, used by `ImapSession` and `SmtpSubmitBackend` directly.
  Moving `BackendConfigField`, `WireLog.Payload` and `TransientRetry` into Contracts (all pure, dependency-free, part of the contract a backend author codes against) lets `Backends.Common` drop the Core reference entirely.
**`Backends.Common` is currently two libraries in one:** converters (~2.1k lines; MimeKit + Ical.Net + FolkerKinzel.VCards) and transport/TLS/wire-logging helpers (~300 lines; MailKit) share no code and have disjoint consumers — `Backends.Local` and `Backends.Dav` want only the converters and pay for MailKit; `Backends.Sieve` uses only `ServerCertificateValidator` + `BackendSchemaFields` and pays for all three converter packages. Split into `Backends.Converters` + `Backends.Transport`. Lower priority than the Core-reference fix.
